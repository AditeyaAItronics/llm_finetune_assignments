S1 · Compute & token budget

Every recent open model overtrains past Chinchilla — the question is by how much

Chinchilla-optimal is 20 tokens/param, tuned only for training FLOPs. Every shipped model ignores it because inference cost dominates lifetime spend. We set our budget by plotting where real models actually sit.

15T tokens. Two real precedents land almost exactly here: Gemma‑3‑27B (14T) and LLaMA‑3‑70B (15T, same corpus as its 8B sibling). At 40B params that's 375 tok/param — 18.75× optimal, deliberately between Gemma‑3‑27B's ratio and LLaMA‑3‑70B's, since a 40B model's deployment lifetime looks more like a 70B's than an 8B's.

Ceiling, not a floor: pushing past ~500 tok/param only helps if the extra tokens are new information, not re-read epochs — which is exactly why §S3 leans on fresh Indic and code data rather than more web text.

S2 · Tokenizer & fertility

Fertility target, per language — then per domain

Coding, science/math, agentic — not measured the same way

These aren't "words," so tokens/word doesn't apply — each gets its own target metric.

3.0–3.3 bytes / token — code

StarCoder2-class compression from the 32K code vocabulary — dense enough that long files still fit in context.

1.0 token / digit — math & science

Per-digit split like Gemma and Qwen (not LLaMA‑3's merged 3-digit tokens) — predictable digit boundaries help arithmetic; LaTeX operators and Greek letters get single dedicated tokens too.

6 reserved tokens — agentic / tool-calling

Dedicated lifecycle tokens (open/close pairs for tool-def, tool-call, tool-result, + a string delimiter) — Gemma 4's real approach — so JSON structure isn't left to generic BPE merges.

256,000-token vocabulary — allocation

English/Latin — 42K Code — 32K Indic scripts — 110K Other languages — 55K Math/agentic/reserved — 17K

256,000 total — matching Gemma‑2's proven size, not Gemma‑3's larger 262,144 (built for Gemini's ~140-language scope, wider than we need). The 110K Indic share exceeds Sarvam‑1's entire 68K vocabulary because we cover all 22 scheduled languages, including low-resource ones Sarvam doesn't attempt.

S3 · Pretraining data mix

Start from LLaMA‑3's disclosed mix, re-weight for the brief

LLaMA‑3 published its actual mix: 50% general, 25% math, 17% code, 8% multilingual. That 8% is right for an English-first model and wrong for ours.

General web — 38% Code — 20% Math/reasoning — 12% Indic (native) — 20% Agentic/synthetic — 5% Other languages — 5%

S4 · Data cleaning

Filters tuned for English quietly break on Indic text

Reused English quality/toxicity/LID tooling doesn't just underperform on Indic languages — it can strip legitimate religious, caste, or regional-political content that gets misread as toxic.

01 · DedupPer-language MinHash/LSHEnglish-tuned thresholds over- or under-merge on Indic scripts.
02 · QualityClassifiers retrained, not portedA translated English quality classifier imports English notions of "good prose."
03 · Language IDCustom LID for code-mixingStandard LID misreads Hinglish/Tanglish — extremely common in real Indian user input.
04 · CodeRepo-level, execution-awareLicense-aware filtering, repo-level dedup, AST validity, lint/execution checks.
05 · PIIIndia-specific patternsStandard PII rules plus Aadhaar-like and PAN identifier formats.
06 · ToxicityThresholds set with native annotatorsPorted English safety classifiers over-censor legitimate Indic discourse.

S5 · Post-training (SFT)

Instructions have to be authored in-language, not translated into it

Machine-translated instructions carry English framing into grammatically-correct Hindi or Tamil — quietly defeating "India first" while looking fine on the page.

S6 · RL / alignment

Verifiable rewards move capability; native annotators move worldview

Only the second lever is where "India first" actually gets encoded — not the pretraining token count.

RLVRCode & mathExecution/unit-test pass rate as reward — language-agnostic, cheap to scale, the DeepSeek/Qwen-class recipe.
Agentic RLMulti-step tool useOutcome-based (task-completion) reward over multi-step tool-use environments, not step-level supervision.
RLHF / RLAIFIndia-based annotatorsPreference data from Indian annotator pools — a US/Euro-centric "constitution" encodes that framing regardless of pretraining data.

Safety red-teaming runs separately across religious, caste, regional, and linguistic axes with Indian annotators — Western safety sets don't cover these.

Open questions we're not pretending are settled

3T-token native Indic web availability is unverified — this assumes synthetic augmentation (backtranslation + native fluency filtering) closes the gap, which is itself unproven at this volume.

The 375 tok/param target assumes genuinely new tokens, not repeated epochs — if Indic scarcity forces epoching, the real ratio is worse than the headline number.
